# Bug Fixes & Optimizations — 2026-03-19

## 1. Radio 重連遞歸調用未 await（潛在並發重連）

**文件**: `lib/services/radio/radio_controller.dart` (line ~830)

**問題**: `_handleStreamEnd()` 在重連失敗時遞歸調用自身，但未使用 `await` 或 `return`。這導致當前調用立即返回，而新的遞歸調用在後台啟動，可能產生多個並發重連嘗試。

**修復**: 改為 `return _handleStreamEnd()`，確保遞歸調用被正確等待，防止並發重連。

---

## 2. 下載服務 `_activeDownloads` 計數器安全性

**文件**: `lib/services/download/download_service.dart` (lines ~425-473)

**問題**: `pauseTask()` 和 `cancelTask()` 中，如果任務同時存在 Isolate 和 CancelToken（理論上不應發生，但防禦性編程），計數器可能被多次遞減導致變為負數。負數計數器會使調度器計算出過多的可用槽位（`availableSlots = maxConcurrent - _activeDownloads`）。

**修復**: 在兩個方法的遞減邏輯後添加 `if (_activeDownloads < 0) _activeDownloads = 0;` 防護。

---

## 3. 歌詞自動匹配並發防護

**文件**: `lib/services/lyrics/lyrics_auto_match_service.dart` (lines ~46-105)

**問題**: `tryAutoMatch()` 沒有防止同一首歌的並發調用。快速切歌時，多個調用可能同時通過「已有匹配」檢查，導致重複 API 請求和潛在的重複數據庫寫入。

**修復**: 添加 `_matchingKeys` Set 追蹤正在匹配的 track key，在方法入口檢查並在 `finally` 中清理。

---

## 5. RankingCacheService 網絡監聽訂閱洩漏防護

**文件**: `lib/services/cache/ranking_cache_service.dart` (line ~98)

**問題**: `setupNetworkMonitoring()` 在 provider 重建時可能創建新訂閱，但舊訂閱已在 `onDispose` 中取消。雖然 `_networkMonitoringSetup` flag 防止了重複訂閱，但為了防禦性編程，在創建新訂閱前應先取消舊的。

**修復**: 在 `listen()` 前添加 `_networkRecoveredSubscription?.cancel()`。

## 7. AudioController `_onTrackCompleted` 異常安全性

**文件**: `lib/services/audio/audio_provider.dart` (line ~2555)

**問題**: `_isHandlingCompletion` flag 在 `Future.microtask()` 外部設為 `true`，但 `logDebug()` 調用也在 microtask 外部。如果 `logDebug()` 拋出異常（理論上不太可能，但防禦性編程），flag 永遠不會被重置為 `false`，導致後續所有歌曲完成事件被永久阻塞。

**修復**: 將 `logDebug()` 移入 microtask 的 `try` 塊內，確保任何異常都會被 `finally` 捕獲並重置 flag。
