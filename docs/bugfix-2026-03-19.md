# Bug Fixes - 2026-03-19

## 1. `_onAudioError` 网络错误重试在 stop() 失败时不触发

**文件**: `lib/services/audio/audio_provider.dart`
**严重程度**: 高

**问题**: `_onAudioError()` 中，当检测到网络错误后调用 `_audioService.stop()`，如果 stop() 抛出异常，`catchError` 只记录日志但不调用 `_scheduleRetry()`。播放器会卡在错误状态，无法自动恢复。

**修复**: 在 `catchError` 中也执行状态重置和重试调度，确保无论 stop() 是否成功，重试机制都能正常工作。

---

## 2. `_executePlayRequest` 错误处理路径中 stop() 未包裹 try-catch

**文件**: `lib/services/audio/audio_provider.dart`
**严重程度**: 中

**问题**: 在 `_executePlayRequest()` 的 catch 块中，网络错误处理和通用错误处理路径都直接 `await _audioService.stop()`。如果 stop() 抛出异常，会跳过 `_scheduleRetry()` 和状态重置，导致 UI 卡在加载状态。

**修复**: 将所有错误处理路径中的 `_audioService.stop()` 包裹在 try-catch 中，确保后续的状态重置和重试调度不受影响。

---

## 3. `MediaKitAudioService.setAudioDevice` 不安全的类型转换

**文件**: `lib/services/audio/media_kit_audio_service.dart`
**严重程度**: 低

**问题**: 使用 `.cast<AudioDevice?>().firstWhere((d) => d!.name == ...)` 模式，虽然运行时安全（因为列表元素实际上非 null），但代码意图不清晰且使用了不必要的强制解包。

**修复**: 改用 `.where((d) => d.name == ...).firstOrNull`，更简洁且类型安全。

---

## 4. `BilibiliSource` 缺少 `dispose()` 方法

**文件**: `lib/data/sources/bilibili_source.dart`
**严重程度**: 低

**问题**: `BilibiliSource` 创建了 `Dio` 实例但没有 `dispose()` 方法来关闭它，与 `YouTubeSource`（有 `dispose()`）不一致。虽然作为应用生命周期单例影响不大，但不符合资源管理最佳实践。

**修复**: 添加 `dispose()` 方法，关闭 `_dio` 实例。
